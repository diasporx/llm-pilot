## Структура проекта

```
llm-pilot/
├── main.py                        — CLI-приложение (точка входа)
├── config.py                      — загружает переменные из .env
├── .env                           — токен, ORG_ID, настройки API (не коммитится)
├── .env.example                   — шаблон для заполнения
├── requirements.txt               — зависимости Python
├── src/
│   ├── client.py                  — HTTP-клиент к Yandex Tracker API
│   └── serializer.py              — сериализация ответов API в JSON
├── data/
│   └── sprint-{N}/
│       └── DEV-XXXXX.json         — выгруженные задачи
├── results/
│   └── sprint-{N}/
│       └── DEV-XXXXX.md           — готовые саммари
├── prompts/
│   └── sprint-summary-prompt.md  — промпт для LLM (единый источник правил генерации)
├── .cursor/rules/summary.mdc     — правила для Cursor AI
├── CLAUDE.md                      — инструкции для Claude Code (загружается автоматически)
└── how-to-use.md                  — этот файл
```

## Первоначальная настройка

### 0. Почему нельзя просто `pip install`?

На Ubuntu/Debian начиная с Python 3.12 система запрещает устанавливать пакеты глобально через `pip`.
Это защита от случайной поломки системных утилит.

**Правило:** всегда устанавливай пакеты только в виртуальное окружение `.venv` внутри проекта.
После создания `.venv` используй не системный `python3`/`pip`, а:
```
.venv/bin/python   — вместо python3
.venv/bin/pip      — вместо pip
```

---

### 1. Установить Python (если не установлен)

Проверь версию:
```bash
python3 --version
```

Нужна версия **3.10 и выше**. Если Python не установлен:
```bash
sudo apt update
sudo apt install python3 python3-venv python3-full
```

> `python3-venv` и `python3-full` обязательны — без них команда `python3 -m venv` может не работать.

---

### 2. Создать виртуальное окружение и установить зависимости

Перейди в папку проекта и выполни **один раз**:
```bash
cd /path/to/llm-pilot
python3 -m venv .venv
.venv/bin/pip install -r requirements.txt
```

Проверь, что всё установилось:
```bash
.venv/bin/python -c "import requests, dotenv; print('OK')"
```

> `.venv` создаётся один раз. После этого достаточно использовать `.venv/bin/python` для всех команд.

---

### 4. Получить OAuth-токен Яндекс Трекера

Перейди по ссылке и авторизуйся:

```
https://oauth.yandex.ru/authorize?response_type=token&client_id=1d0b9dd4d652455a9eb710d450ff456a
```

После авторизации в URL появится `access_token=...` — скопируй его.

### 5. Заполнить .env

Скопируй шаблон и вставь свои данные:

```bash
cp .env.example .env
```

Открой `.env` и заполни:

```env
TRACKER_TOKEN=y0_AgAAAA...   # OAuth-токен из шага 2
ORG_ID=XXXXXXX               # ID организации (см. ниже)
ORG_TYPE=yandex360           # yandex360 или cloud
BASE_URL=https://api.tracker.yandex.net/v2
```

**Как найти ORG_ID:** запусти `whoami` (см. ниже) — он найдёт ID автоматически.
Либо вручную: зайди на `https://admin.yandex.ru` — в URL будет `company/XXXXXXX`, где `XXXXXXX` и есть `ORG_ID`.

> `.env` добавлен в `.gitignore` и не попадёт в репозиторий.

### 6. Проверить подключение

```bash
.venv/bin/python main.py whoami
```

Ожидаемый результат:
```
Токен валиден.

Найдены организации Яндекс 360:
  ORG_ID = "XXXXXXX"  —  Название компании
```

## CLI — команды для выгрузки задач

Все команды запускаются через `.venv/bin/python main.py`.
Не используй системный `python3` — нужен именно Python из виртуального окружения.

---

### `whoami` — проверка токена и организации

```bash
.venv/bin/python main.py whoami
```

Проверяет, что токен валиден, и выводит список доступных организаций с их ORG_ID.

---

### `issue` — выгрузить одну задачу

```bash
.venv/bin/python main.py issue DEV-18161
```

Если задача привязана к спринту — программа определит его автоматически и сохранит в `data/sprint-{N}/DEV-18161.json`.

**Если задача не привязана к спринту** — укажи номер вручную:

```bash
.venv/bin/python main.py issue DEV-18161 --sprint 31
```

Также принимает полное название спринта:

```bash
.venv/bin/python main.py issue DEV-18161 --sprint "Спринт 31 ПП"
```

---

### `sprint` — выгрузить весь спринт

```bash
.venv/bin/python main.py sprint --sprint "Спринт 31 ПП" --queue DEV
```

- `--sprint` — точное название спринта, как оно называется в Трекере
- `--queue` — очередь задач (например `DEV`)

Программа выгрузит все задачи спринта и сохранит их в `data/sprint-31/`.

Пример вывода:
```
Fetching sprint 'Спринт 31 ПП', queue 'DEV'...
Found 29 issues

  DEV-18495  [Баг] Некорректное отображение...
  DEV-17525  [Разработка] Реализация...
  ...

Saved 29 files → data/sprint-31/
```

## Генерация саммари через LLM

После выгрузки JSON-файлов саммари генерирует LLM — **Claude Code** или **Cursor AI**.
Промпт с правилами генерации находится в `prompts/sprint-summary-prompt.md`.

### Через Claude Code

Просто напиши в чате запрос на русском языке. Claude сам определит режим работы.

**Саммари одной задачи:**
```
Сделай саммари по DEV-18161
```
```
Разбери задачу DEV-16402
```

Если JSON-файла нет — Claude выгрузит задачу из Трекера автоматически и спросит номер спринта, если не сможет определить его сам.

**Саммари всего спринта:**
```
Сделай саммари по всем задачам 31 спринта
```
```
Напиши саммари для каждой задачи спринта 28
```

Если папки `data/sprint-{N}/` нет — Claude предложит сначала выгрузить спринт.

### Через Cursor AI

Правила загружаются автоматически из `.cursor/rules/summary.mdc` (флаг `alwaysApply: true`).
Запросы те же, что и для Claude Code — просто напиши в чате.

## Структура саммари

Каждое саммари — это мини-ТЗ для ревью кода или постановки задачи тестировщику.
Файлы сохраняются в `results/sprint-{N}/DEV-XXXXX.md`.

```markdown
# DEV-XXXXX: Название задачи

https://tracker.yandex.ru/DEV-XXXXX

**Тип:** Task | **Статус:** Решен | **Исполнитель:** Имя Фамилия

---

## Описание

Что за задача, зачем нужна, какую проблему решает.

---

## Договорённости и комментарии

Содержательные решения из обсуждения: уточнения требований,
изменения scope, согласованные технические подходы.
Если ничего содержательного нет — «Нет».

---

## Для разработчика

Что реализовано / изменено. Ветки, MR, эксперименты, затронутые модули.

---

## Для тестировщика

Что и как проверять: пользовательский путь, граничные случаи, DoD-критерии.
```

**Важно:** в комментарии попадает только то, что меняет требования или добавляет технический контекст.
Фразы «передал на ревью», «без замечаний», «ок» — не включаются.

## Типичные сценарии использования

### Сценарий 1: Новый спринт закончился, нужны саммари

```bash
# 1. Выгрузить все задачи спринта
.venv/bin/python main.py sprint --sprint "Спринт 32 ПП" --queue DEV

# 2. Написать LLM:
# "Сделай саммари по всем задачам 32 спринта"
```

Результат: `results/sprint-32/DEV-XXXXX.md` для каждой задачи.

---

### Сценарий 2: Нужно саммари по одной задаче

Задача уже была выгружена ранее:
```
Сделай саммари по DEV-16402
```
LLM найдёт файл в `data/sprint-*/DEV-16402.json` и создаст саммари.

Задача ещё не выгружена:
```
Сделай саммари по DEV-16402
```
LLM выгрузит задачу сам (или спросит номер спринта), а затем создаст саммари.

---

### Сценарий 3: Переделать саммари по новым правилам

```
Переделай саммари по 31 спринту
```

LLM перечитает JSON-файлы из `data/sprint-31/` и перезапишет все `.md` в `results/sprint-31/`.

---

### Сценарий 4: Задача не привязана к спринту

```bash
.venv/bin/python main.py issue DEV-16402 --sprint 30
```

Файл сохранится в `data/sprint-30/DEV-16402.json`.

## Решение проблем

### Ошибка: `Токен недействителен`

Токен истёк. Получи новый:
```
https://oauth.yandex.ru/authorize?response_type=token&client_id=1d0b9dd4d652455a9eb710d450ff456a
```
Вставь новый токен в `.env` → `TRACKER_TOKEN`.

---

### Ошибка: `HTTP 403` при выгрузке

Неверный `ORG_ID` или `ORG_TYPE`. Запусти `whoami` — он покажет правильный ID.
Убедись, что `ORG_TYPE` соответствует типу организации (`yandex360` или `cloud`).

---

### Ошибка: `Спринт не указан и не найден в задаче`

Задача не привязана к спринту в Трекере. Укажи вручную:
```bash
.venv/bin/python main.py issue DEV-XXXXX --sprint 31
```

---

### Файл JSON слишком большой для чтения LLM

Некоторые задачи с большим количеством комментариев могут превышать лимиты контекста.
LLM в этом случае использует `Grep` для извлечения ключевых полей — это нормальное поведение.

---

### Название спринта не найдено (`Found 0 issues`)

Название спринта должно точно совпадать с тем, как оно называется в Яндекс Трекере.
Проверь название в интерфейсе Трекера и передай его в кавычках:
```bash
.venv/bin/python main.py sprint --sprint "Спринт 31 ПП" --queue DEV
```
