# Инструкция: генерация саммари задач

## Определи режим работы

Перед выполнением определи, что просит пользователь:

---

### Режим 1 — Саммари одной задачи

**Триггер:** пользователь упоминает конкретный ключ задачи.
Примеры: _«сделай саммари по DEV-18161»_, _«мне нужно саммари DEV-17999»_, _«разбери задачу DEV-18161»_.

**Шаги:**

1. Найди файл `data/sprint-*/[KEY].json` (поиск по всем спринтам).
2. Если файл не найден — выгрузи задачу из Яндекс Трекера:
   ```bash
   .venv/bin/python main.py issue [KEY]
   ```
   - Если программа вернула ошибку «Спринт не указан» — спроси у пользователя номер спринта и повтори с `--sprint [N]`.
   - Если ошибка токена (401 / expired_token) — сообщи, что нужно обновить `TRACKER_TOKEN` в `config.py`.
3. Прочитай JSON-файл.
4. Создай саммари по структуре ниже.
5. Сохрани в `results/sprint-[N]/[KEY].md`.
6. Сообщи пользователю путь к созданному файлу.

---

### Режим 1 — дополнительное правило: пропуск готовых

Перед шагом 3 проверь: существует ли уже `results/sprint-[N]/[KEY].md`.
Если файл **существует** — не читай JSON, не генерируй саммари, сообщи пользователю «уже есть» и остановись.

---

### Режим 2 — Саммари всего спринта

**Триггер:** пользователь называет номер спринта без конкретного ключа задачи.
Примеры: _«напиши саммари по 31 спринту»_, _«сделай саммари для каждой задачи 25 спринта»_, _«обработай все задачи спринта 31»_.

**Шаги:**

1. Найди все файлы `data/sprint-[N]/*.json`.
2. Если папка не найдена — предложи сначала выгрузить спринт:
   ```bash
   .venv/bin/python main.py sprint --sprint "Спринт [N] ПП" --queue DEV
   ```
3. Отфильтруй задачи: **пропусти** те, для которых уже существует `results/sprint-[N]/[KEY].md`.
4. Для каждой **оставшейся** задачи запусти отдельный **Agent (subagent)**, передав ему только путь к JSON-файлу и инструкцию создать саммари. Это изолирует контекст каждой задачи.
5. После завершения выведи список всех созданных файлов (и список пропущенных как уже готовых).

---

## Назначение саммари

Саммари — мини-ТЗ для ревью: LLM или разработчик должны понять суть изменений, не открывая трекер.
Пиши кратко и по делу. **Не включай** фразы-пустышки: «передал на ревью», «без замечаний», «замечаний нет», «всё хорошо», «посмотри пожалуйста».
Из комментариев бери только то, что меняет требования, фиксирует решение или добавляет технический контекст.
Пиши на **русском языке**. Не придумывай детали, которых нет в файле.

---

## Структура выходного файла

```markdown
# [KEY]: [summary из JSON]

**Тип:** [type] | **Статус:** [status] | **Исполнитель:** [assignee]

---

## Описание

Что за задача, зачем нужна, какую проблему решает.
Источник: поле description (блок «Описание» или «Основное»).

---

## Договорённости и комментарии

Только содержательные решения: уточнения требований, изменения scope, согласованные подходы.
Если ничего содержательного нет — написать: «Нет».

---

## Для разработчика

Что реализовано / изменено. Какие модули, сервисы, ветки, MR, эксперименты задействованы.
Архитектурные решения, если зафиксированы.

---

## Для тестировщика

Что и как проверять: пользовательский путь, граничные случаи, DoD-критерии.
Для операционных задач (раскатка FF, выгрузка данных) — что считается успешным результатом.
```
